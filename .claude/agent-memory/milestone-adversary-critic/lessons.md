## graph-and-window-2026q2-e1 (2026-05-21)
- AI-3 analysis: vanilla QMainWindow (no QtInteractor) is safe under offscreen per AI-3's one-line rule. The ban is specifically on MainWindow (the app's class which hosts QtInteractor). Check zero-QtInteractor condition explicitly when a vanilla QMainWindow appears in a diff.
- Tool-parity gap pattern: when a milestone adds lighting params to app.py (e.g., ambient/diffuse), check whether the visual-scout template in agent-prompts.md was also updated to match. UPL-9 fixed the live app but the scout template only got the background fix (UPL-28), missing the lighting delta. Grep: `grep -A3 "add_mesh" .claude/references/frontend-uplift/agent-prompts.md` and compare params against app.py's add_mesh call.
- QDockWidget ownership: QDockWidget.setWidget() transfers C++ ownership. QDockWidget has no takeWidget() (unlike QMainWindow.takeCentralWidget()). The correct workaround is panel.setParent(None) in a finally block before the host goes out of scope. If this pattern appears, flag absence from CONTEXT.md §8 as a LOW.
- Severity calibration: a 241-LOC tooling/script diff (no new generators, no new ParamSpecs, no new surfaces) is unlikely to produce more than 1 MEDIUM. Axis 7 (test coverage) and Axis 1 (AI invariants) are both N/A for render-script changes — concentrate scan on Axis 8 (offscreen correctness) and Axis 5 (color token discipline).

## variety-palette-2026q2-e1 (2026-05-21)
- Stale forward-reference pattern: when a milestone fills an existing stub, the FILLING commit must also update any "will populate" forward-reference comments elsewhere in the file (module docstring, block comments, __init__ comments). grep `"will populate\|will override\|will add"` in all touched files before declaring the diff complete.
- Test coverage gap for new public methods on Qt widgets: palette-level data-dict tests do NOT cover the method that consumes the dict. Even if a Qt widget can't be fully instantiated in tests (AI-2), a test that monkey-patches the widget-update helper and inspects the stored state directly is achievable and should be required for new public APIs.
- Duplicate test helper pattern: extracting a nested helper to module-level while leaving the original in place "for self-contained readability" is a scope-discipline violation (Axis 10: backwards-compat shims). The correct action is to refactor the original call site to use the module-level helper. Flag this as MEDIUM even when both implementations are byte-for-byte identical today.
- Unicode key audit: `python3 -c "with open(f,'rb') as f: ..."` or the codepoint-per-char loop approach is the surer check than visual inspection. Record this as the standard for future milestones touching styles.py keys.
- Wire-path analysis for "set-then-render" patterns: when a method updates internal state without triggering a render, verify the caller chain ensures a render WILL happen (on the same user action or the next one). In this case set_default_color is called in _on_variety_changed (no render) and again in _on_subtype_changed (which DOES call _render_current). The double-call is harmless but the first call only updates the swatch preview — document this semantic clearly.
