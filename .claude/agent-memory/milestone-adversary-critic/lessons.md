## realtime-variety-render-e5b (2026-05-23)
- Test-reference verbatim contract: when a test docstring says "reference reproduced verbatim from pre-JIT generator body," grep the reference function for `**5` or `**3` operators vs the original source (git show prior-commit:surfaces.py). If the reference has been adapted to use explicit multiply chains (matching the kernel), the "verbatim" claim is false even if tests pass (atol=1e-9 bridges the ULP gap). The research brief for this milestone explicitly required `X**5` in the reference; the implementation silently inverted to `x2*x2*x`. This is the dominant failure mode for kernel-transcription milestones: both the kernel AND the reference use explicit chains, so the test correctly passes but provides weaker independence guarantees than claimed.
- IEEE-754 mismatch diagnostic: `x**3 != x*x*x` for ~50% of random float64 inputs (ULP diffs 5e-17 to 4e-15). `x**5 != x2*x2*x` similarly. At `atol=1e-9` for polynomial field values, these ULPs are always within tolerance — but the independence guarantee is destroyed. Always check by running `python -c "import numpy as np; x=np.float64(1.7); print(x**5 == x*x*x*x*x)"` — a False means the reference is not truly independent.
- Single-commit violation: when an implementation plan specifies "≤200 LOC each — likely N commits," and the implementer ships as a single commit, log as MEDIUM (not HIGH) — it's a process violation, not a correctness bug. The fix is a process note, not a code change.
- Axis-transposition partial coverage: sextic double solid is asymmetric in z vs x/y (z^2 vs x^6) so a k<->i or k<->j swap IS caught, but an i<->j (x<->y) swap is NOT caught at alpha=0 (field symmetric in x,y). The docstring saying "would surface an axis-swap bug" is technically correct for the x<->z swap but misleading about the x<->y case. Flag as LOW documentation precision.
