## variety-palette-2026q2-e1 (2026-05-21)
- Stale forward-reference pattern: when a milestone fills an existing stub, the FILLING commit must also update any "will populate" forward-reference comments elsewhere in the file (module docstring, block comments, __init__ comments). grep `"will populate\|will override\|will add"` in all touched files before declaring the diff complete.
- Test coverage gap for new public methods on Qt widgets: palette-level data-dict tests do NOT cover the method that consumes the dict. Even if a Qt widget can't be fully instantiated in tests (AI-2), a test that monkey-patches the widget-update helper and inspects the stored state directly is achievable and should be required for new public APIs.
- Duplicate test helper pattern: extracting a nested helper to module-level while leaving the original in place "for self-contained readability" is a scope-discipline violation (Axis 10: backwards-compat shims). The correct action is to refactor the original call site to use the module-level helper. Flag this as MEDIUM even when both implementations are byte-for-byte identical today.
- Unicode key audit: `python3 -c "with open(f,'rb') as f: ..."` or the codepoint-per-char loop approach is the surer check than visual inspection. Record this as the standard for future milestones touching styles.py keys.
- Wire-path analysis for "set-then-render" patterns: when a method updates internal state without triggering a render, verify the caller chain ensures a render WILL happen (on the same user action or the next one). In this case set_default_color is called in _on_variety_changed (no render) and again in _on_subtype_changed (which DOES call _render_current). The double-call is harmless but the first call only updates the swatch preview — document this semantic clearly.

## dark-mode-2026q2-e1 (2026-05-21)
- Inline-style override pattern: widget.setStyleSheet() with an explicit color: property OVERRIDES the QApplication-level stylesheet. When a milestone adds dark-mode via QApplication.setStyleSheet, any pre-existing inline setStyleSheet calls on individual widgets remain hardcoded to the old (light) palette. Grep: `grep -rn "setStyleSheet(.*_STYLE\|setStyleSheet(.*COLOR_\|setStyleSheet(.*color:" *.py` — every match using a light-mode-only constant is a dark-mode contrast failure. This is the dominant failure mode for "QSS-swap" dark mode implementations.
- WCAG verification: all researcher-claimed PALETTE_DARK ratios verified independently and matched to 2 decimal places. The pattern of "researcher cites ratio in comment; verify with WCAG 2.x relative luminance" is the correct review move and adds ~5 min of computation per palette.
- AI-11 calibration: Qt.ColorScheme.Light is the CORRECT fully-qualified enum form (the enum class lives on Qt namespace directly); this is NOT an AI-11 violation. Compare: Qt.AlignLeft (bad, should be Qt.AlignmentFlag.AlignLeft) vs Qt.ColorScheme.Light (correct — ColorScheme IS the direct enum class).
- PySide6 signal.connect() returns QMetaObject.Connection; signal.disconnect(conn_obj) IS valid PySide6 idiom. Confirmed in project venv. No bug in the signal lifecycle management pattern.
- Deferred-inline-style documentation gap: when a dark-mode milestone is V0 scoped to QSS-only, the known limitation (inline style constants remain light-palette-only) MUST be documented in CONTEXT.md as a named V0 gap — otherwise future milestone authors see 192 green tests and assume dark mode is complete.

## enriques-backface-2026q2-e1 (2026-05-22)
- Topology-claim precision gap: when a milestone applies a rendering fix justified by "topology class X" (e.g., double-curve singularities), verify the claim holds for ALL subtypes in the targeted variety family, not just the primary figure. The Cayley symmetroid (Fig. 3) is a degree-4 surface with ordinary nodes, not double-curve; the sextic claim applied to Figs. 1 and 2 only. When identical bytes appear in render validation, this confirms the fix is harmless but also suggests the stated rationale may not apply — flag as a MEDIUM documentation inaccuracy.
- PyVista Property.culling API: accepts "back", "front", "none" as strings; raises ValueError for any other value including Python None. The `value or "none"` idiom correctly maps Python-None to the "none" off-state. Verified in project venv. No AI-13 issue (not a color value).
- Pattern-A subtype-switch analysis: when a variety-level setter (like set_culling) is NOT wired into _on_subtype_changed, the correct question is "should the state change on subtype switch?" For culling, NO — all Enriques subtypes share the culling policy, so persistence across subtype switches is the intended semantic. Only flag this gap if the state SHOULD change on subtype switch (which set_default_color does, calling it in both _on_variety_changed and _on_subtype_changed).
- Severity calibration: an 81-insertion-only diff with a single MEDIUM documentation inaccuracy is a correctly calibrated result. No inflation needed. The absence of CRITICALs and HIGHs is accurate given the per-variety gate is sound and all AI invariants are respected.
