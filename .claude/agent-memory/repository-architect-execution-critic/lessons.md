
## Lesson from restructure-full-audit-2026q2-r1 (2026-05-23)

### New axis worth adding
- **Intermediate-commit greenness is the most important bisect check for multi-op batches.** When a batch has N `git mv` commits followed by a LibCST import-rewrite commit, the intermediate mv commits will break any test that imports the moved module through an unrewritten importer (e.g., `app.py` still has old imports; a test that imports `from app import X` will fail with `ModuleNotFoundError`). Always check whether the test suite can survive each `git mv` commit independently BEFORE the LibCST rewrite lands. The safe pattern is either: (a) `git mv` + LibCST rewrite in the same commit, or (b) pre-write shims before any `git mv`.

### False-positive patterns to avoid
- **`validate-shims.py` FAIL on Template-2 `__getattr__` shims is always a false negative.** The script runs `import <module>` which does NOT trigger `__getattr__`; only attribute access does. Don't escalate this to CRITICAL or HIGH. The correct check is: do all 4 pytest shim tests pass? Do the manual smoke-tests (`from <old_mod> import <Symbol>`) emit DeprecationWarning? If yes to both, the shims are correct and the script is the problem.
- **`verify-anchors.py` reporting 400+ "stale" anchors after a move that includes `parameter_grid_panel` in a path is a known false positive.** The `parameter_grid_panel` needle matches `panels/parameter_grid_panel.py` (the correct new path). This is a substring match bug in the script. Don't elevate the 400+ count to a CRITICAL unless genuine stale paths remain in CLAUDE.md, AGENTS.md, CONTEXT.md, or README.md.

### AVC-specific post-execution gotchas
- **`pydeps` cycle check is structurally impossible for this repo** (directory name `algebraic-variety-cross-section` is not a valid Python identifier; pydeps rejects it). Rubric item 6 must be verified via `python -c "import app"` smoke-test instead.
- **`coverage xml` fails due to pyvistaqt internal path error** in this repo. Rubric items 4 and 5 cannot be mechanically checked until this is fixed. This is pre-existing, not introduced by any restructure.
- **PLAN "~9 total commits" is consistently an undercount for 4-batch restructures.** Actual commit counts are closer to 18–24 (source) + 4 (pipeline metadata). Calibrate user expectations accordingly.
- **stacklevel=2 in shims correctly points at the caller.** Verified by running the shim with `-W always` and checking the file:line in the warning output — it shows `<string>:N` (the calling code's line), not the shim's internal line.

## Lesson from restructure-single-root-2026q2-r3 (2026-05-24)

- **Gate-command auditability:** When a batch log reports "gate passed: exit 0" for a mandatory inter-commit checkpoint (e.g., `-W error::DeprecationWarning`), the VERBATIM COMMAND must appear in the log, not just the result output. Without it, future auditors cannot confirm whether the full `-W error` invocation was actually run or just `pytest -q`. Add this to the implementer template: "gate section must log verbatim command AND result."
- **In-module backward-compat docstrings are NOT in the anchor-updater's scope.** The anchor-updater updates CLAUDE.md/README.md/MOVES.md/CONTEXT.md but does not scan source files for docstring claims. When a hub module is deleted (e.g., surfaces.py), the sub-modules it re-exported (varieties/registry.py, varieties/tooltips.py) may still contain "still works via hub-shim" docstring lines that become factually false. These must be flagged as a MEDIUM finding and cleaned as a post-deletion step.

- **A `python -W error::DeprecationWarning -m pytest -q` run returning 0 warnings does NOT mean the gate was skipped.** If the hub module being retired was a pure re-export shim (no `__getattr__` DeprecationWarning), the gate still has the correct effect: it verifies no r2 Qt shim is still active. The gate is vacuously correct in this context. Do not escalate to HIGH just because the shim didn't itself emit DeprecationWarning — check what the gate was actually testing.
- **Root `__pycache__` stale pyc files for deleted modules are LOW severity, not CRITICAL.** Python import machinery resolves against live `.py` files first; stale `.pyc` files for deleted modules do not cause test failures or import errors. Flag as LOW rubric item 19 failure, not CRITICAL.
- **`refactor-r3-b5-end` tag pointing to the penultimate commit (not the final SHA-finalization commit) is LOW severity.** When the anchor-updater requires a two-step docs commit (main docs + SHA finalization), the batch-end tag is typically placed after the main docs commit. The 1-line SHA-finalization commit is effectively zero-impact on rollback correctness. Do not escalate to HIGH.

- **String-literal scanner gap is a recurring r3 pattern.** `test_icons.py` used `importlib.import_module("icons")` (a string literal) which LibCST cannot rewrite — caught and fixed manually in B3. This is the same "path-string scanner" pattern from r2 L4. Any future batch that deletes a shim should grep for `importlib.import_module("<shim_name>")` and `sys.modules.pop("<shim_name>")` patterns in test files BEFORE declaring the batch clean.
- **ScopeProvider omission from METADATA_DEPENDENCIES is safe.** PLAN.md specified `METADATA_DEPENDENCIES = (QualifiedNameProvider, ScopeProvider)` but B1 omitted ScopeProvider (unused). This is not a bug — adding unused metadata providers increases parse time. Accept the deviation.
- **r3 commit count of 11 (9 source + 2 docs) is on the low end of the ~13 prediction.** The shorter count reflects fewer metadata/checkpoint commits than r1/r2 (those had 18-24 source commits). r3 was genuinely simpler: B3 was 1 commit, not 6. Calibrate: "~13 commits" is reliable for r3-class restructures (1 move + 5 deletes + 1 retire + lock-in).
