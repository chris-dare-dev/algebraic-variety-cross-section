
## Lesson from restructure-full-audit-2026q2-r1 (2026-05-23)

- Anti-pattern caught that escaped audit: A partial monolith extraction (Batch 5 kernels: 401 LOC of 1811 → leaves 1410 LOC, closes zero evaluator FAILs) was dressed up as "addressing FAIL #17" when the current-state auditor's own §13 said "don't fix" for this batch. The audit briefs can contain self-contradictory signals (§4.2 says "cleanest seam" while §13 says "already well-organized, no urgent extraction"). The design-adversary should explicitly cross-check §4 recommendations against §13 "already good" entries before endorsing an extraction.

- Anti-pattern caught that escaped audit: Naming a re-export "NOT a deprecation shim" to avoid the shim lifecycle. For private _underscore symbols the DeprecationWarning cycle is legitimately skippable (R10 exception), but the symbol still needs a MOVES.md entry so future maintainers know the authoritative location. The distinction "permanent re-export vs shim" only matters for the warning; the MOVES.md tracking obligation survives it.

- Anti-pattern caught that escaped audit: Tier-3 rollback using `git log --grep` on an uncommitted commit message pattern. This is fragile — the grep pattern must match exactly, and commit message formats drift. Prefer tag-based Tier-3 rollback: tag after each batch, revert from tag.

- New axis worth adding to the 11: "Partial extraction audit" — for any Extract Module operation, check: (a) does the extraction close at least one evaluator FAIL or fix an invariant violation, or (b) does it reduce the source file below a stated LOC threshold? If neither, the extraction is premature optimization within the conservative-bias brief.

- AI-invariant subtlety: AI-6 (Numba threading layer) has an import-order dependency that becomes a hidden footgun when kernels are extracted to a private module. The config write (`numba.config.THREADING_LAYER = "workqueue"`) must precede `from numba import njit` in the module that houses the kernels — not just in the module that imports the kernel module. This is self-healing if you mirror the surfaces.py pattern but is easy to miss if the implementer just moves the kernel functions and adds a re-export without also moving the config preamble.

- MOVES.md creation: always listed in Phase 8, easy to omit from tree-diff in Phase 2. The design-adversary should check that `+ MOVES.md` appears in the tree diff for the first batch that contains file moves.

## Lesson from restructure-feature-subpackages-2026q2-r2 (2026-05-23)

- Anti-pattern caught that escaped audit: The PLAN claimed "no callers of `from panels.X import Y` other than app.py" and used this as the stated rationale for omitting a `panels/__init__.py` hub shim. Actual grep of the repo revealed TWO test files (`tests/test_clip_domain.py:21` and `tests/test_styles_palette.py:243,280,301,320`) import from `panels.*` canonically. The designer's grep verification was silently wrong (grep filtered `*.py` but may have excluded the test/ subdirectory, or the pattern `from panels\\|import panels` missed `import panels.appearance`). Lesson: the design-adversary MUST independently run the grep rather than trust the PLAN's "Verified" claim. Always check test files for callers — test files are the most common overlooked callers for internal package renames.

- Anti-pattern caught that escaped audit: Symbol `FAST_RENDER_THRESHOLD_MS` (defined at `surfaces.py:96`, imported in `tests/test_typical_ms.py:25`) was absent from both symbol-map.json and the hub-shim `_PUBLIC_NAMES` dict, despite being tested as a HARD lock in the current-state brief. The refactor-pattern scout's sample `_PUBLIC_NAMES` included it, but the designer's final PLAN did not propagate it. Lesson: the design-adversary should diff the current-state brief's "critical path locks" table against the symbol-map.json to find missing symbols. Constant symbols co-located with dispatch functions are easy to miss.

- Anti-pattern caught: A tree-diff annotation contained a contradictory placement for `_hanson_cross_section` — line 101 put it in `varieties/_marching.py` and line 108 put it as a "local helper" in `varieties/calabi_yau.py`. The symbol-map correctly resolved it, but the implementer reads the tree diff, not the symbol map, for file-content guidance. Lesson: the design-adversary should cross-check every symbol that appears in BOTH the tree diff AND the symbol map, and flag any inconsistency.

- New axis observation: The shim-test deletion / deferral pattern is a recurring rationalization. The R1 anti-pattern "we can refactor and add features in the same PR" was misapplied to shim tests — which are regression guards, not features. The design-adversary should specifically check whether the PLAN defers shim tests by invoking R1, and flag it as MEDIUM if so. Shim tests are not features; they are the parity verification the refactor-pattern brief prescribes in §5.7 and §7 (Phase 7).

- AI-invariant subtlety (Numba threading layer): `varieties/__init__.py` must contain `import varieties._kernels` as an eager import to guarantee the process-global `numba.config.THREADING_LAYER = "workqueue"` side effect fires when `import varieties` runs. A lazy-loading `__init__.py` that does NOT eagerly import `_kernels` will delay the side effect until a kernel function is first accessed — which is safe for interactive use but is later than the current behavior and may fail test/CI patterns that check the threading layer state before calling any generator.

## Lesson from restructure-single-root-2026q2-r3 (2026-05-24)

- Anti-pattern caught that escaped audit: The current-state auditor's §6 B4 budget table counted grep hits including docstring-only mentions (e.g. `varieties/tooltips.py:13` and `varieties/registry.py:13` contain `from surfaces import` in their docstring strings). This inflated the file count from 16 live files to 19 and the import-line count from ~20 to 23. The design-adversary MUST independently verify file counts with `grep -rln` and then manually confirm that each hit is a live import statement (not a docstring example), not trust the audit's grep-derived count.

- Anti-pattern caught: A PLAN's axis-12 self-check can claim "no audit brief used stronger-than-neutral language" when in fact the best-practices scout DID use a trigger phrase ("primary monolith risk" for app.py at 1900 LOC) — but the same sentence in the scout endorsed the deferral. The design-adversary should grep audit briefs for ALL axis-12 trigger words, then check whether the scout's surrounding context endorses or condemns deferral. A scout that uses a trigger word AND endorses deferral in the same sentence is self-evidently not in conflict with the PLAN; the self-check claim is imprecise but not wrong.

- Anti-pattern caught: A PLAN can silently contradict a scout brief about smoke-test parametrize scope (scout: "DO NOT include app/_qt"; PLAN: "parametrize over app, _qt") when the specific app structure makes the scout's concern inapplicable (app.py wraps QApplication in main(), not at module scope; _qt/__init__.py is docstring-only). The design-adversary should verify whether app.py constructs QApplication at module scope before citing the scout's "do not include app" guidance as a finding. A subprocess-based smoke test that runs `import app` is safe when app.py's QApplication is inside `if __name__ == "__main__"`.

- New axis observation: Batch step descriptions that list (a)(b)(c)(d)(e) sub-steps without mapping each to "commit" vs "verification checkpoint" are ambiguous enough to cause an implementer to collapse all steps into a single commit, eliminating intermediate green checkpoints. For HIGH-risk batches, the design-adversary should verify that the commit mapping is explicit, not implied.

- AI-invariant subtlety: The refactor-pattern-scout's "DO NOT include app in subprocess smoke tests" guidance is conditional on whether app.py constructs QApplication at module scope. AVC's app.py does NOT — QApplication is inside main() which is called under `if __name__ == "__main__"`. Future designs including `app` in subprocess smoke tests are AI-2 compliant for AVC specifically.
