## enriques-backface-2026q2-e1 (2026-05-22)
- For backface-culling milestones, ALWAYS run per-family off-screen renders before declaring "XS safe." The Enriques surface has open-sheet double-curve geometry (visible from both sides); Hanson has AI-7 winding-per-patch normals. Both break catastrophically under global culling='back'. The Fermat quartic (closed convex) is the only family where global culling is safe.
- The Hanson AI-7 conflict with culling is confirmed by render: 50% of cells face away from any camera position (expected for a sphere-like ball of patches), so culling='back' hides entire far-side patches on rotation. This is a load-bearing reason to never apply global backface culling to parametric disconnected-patch surfaces.
- The Enriques canonical sextic `enriques_figure_1` default is c=1.0 (min 0.1 per ParamSpec), NOT c=0. c=0 raises ValueError (no real zero set — field range [9.65e-09, 10]). Any milestone brief referencing "Enriques default c=0" contains a documentation error.
- Per-family actor property gating belongs in `AppearancePanel.apply_to_actor()` via a stored override field (e.g. `self._culling = None`), set by a `set_culling(value)` method called from `_on_variety_changed` / `_on_subtype_changed` in app.py. This keeps all actor-property assignments in the canonical architectural home.
- Challenger NONE ratings for single-kwarg rendering changes can be wrong when the kwarg's effect depends on per-family mesh topology (closed vs. open-sheet vs. disconnected patches). Always render first.
- PyVista `culling=True` is equivalent to `culling='back'` — confirmed via `actor.prop.culling == 'back'` check after `add_mesh(mesh, culling=True)`. Default is `culling=None` which maps to `actor.prop.culling == 'none'`.

- For VTK/PyVista pipeline-replacement milestones, run small `.venv/Scripts/python.exe -c "..."` probes first — they resolve disputed API questions in <30 seconds and eliminate speculation entirely.
- **Flying Edges compute_normals**: `pv.ImageData.contour([level], method='flying_edges', compute_normals=True)` produces gradient-based normals in the `Normals` point array (unit-length, verified probe). Challenger "normals lost" objections against Flying Edges are moot when this kwarg is used. The `compute_normals` parameter is accepted by PyVista 0.48.4 + VTK 9.6.2.
- **Ravel-order critical bug**: `indexing='ij'` meshgrid + `pv.ImageData` requires `field.ravel(order='F')` (Fortran/column-major). Always verify with an asymmetric ellipsoid (x_radius≠z_radius) probe before declaring the implementation correct.

The Kitware "1–2 orders of magnitude" benchmark is multi-thread Intel x86; the single-thread Windows 11 reality is 7–10x on the MC step but the full pipeline includes field-eval (not accelerated by FE) and Taubin/normals, diluting to ~2.5x overall.
- For S-effort pipeline-replacement milestones where the implementation path is well-specified, the agent-b brief (VTK/PyVista lens) is more valuable than the agent-a brief (math lens) — the implementation path is all API surface.
