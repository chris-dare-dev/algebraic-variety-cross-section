# Frontend-Uplift Challenger lessons

- **QSettings persistence is routinely undercounted as "M" when it is "M+"**: the API surface looks small but the signal-cascade re-entrancy risk on restore (variety-changed → subtype-changed → render fires N times), schema versioning (namespace + integer version tag), and key-existence guard for renamed varieties (must check `if key not in VARIETIES`) together add 1–2 extra days beyond the QSettings API calls themselves. Size QSettings candidates at the upper-end of M, not lower-end.
- **superqt throttled signals are an INT-NO-1 anti-pattern for ≥500 ms render callbacks**: `superqt.signals.throttled` is a leading-edge throttle that fires on first event and suppresses for N ms; for a marching-cubes render (~500 ms), it can fire multiple renders during a single drag if the throttle interval is less than the render time. The correct primitive for a slow render pipeline is `sliderReleased` (fires once on mouse-up), not throttle. Always verify the render-callback latency before proposing a throttle replacement.
- **Animation candidates must explicitly set `self._computing = True` for the duration of any direct `plotter.render()` tick loop**: camera-animation and parameter-sweep candidates that bypass `_render_current` to call `plotter.render()` directly avoid re-entrancy but leave `_computing = False`, allowing a concurrent slider-release to enter `_render_current` during a VTK render cycle. The fix pattern is: set `_computing = True` at animation start, clear in `finished` signal.
