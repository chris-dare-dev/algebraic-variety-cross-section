# Competitive Scout — Generalizable Lessons

- **Two-pass LOD is the universal peer pattern for slow parameter-driven rendering**: every tool surveyed (Mathematica `ControlActive`, ParaView interactive/still render, Desmos web worker) converges on coarse-preview-during-drag + full-quality-on-release; when a peer app claims "real-time" algebraic surface response it is almost always achieving this via one of: (a) GPU raytracing (SURFER — no tessellation step), (b) flying-edges-class parallelized MC, or (c) reduced-resolution preview during drag. The INT-2 "release-only" discipline is correct for a blocking main-thread render, but the missing half is the coarse preview path.
- **`pv.ImageData.contour(method='flying_edges')` is the drop-in MC replacement in PyVista**: when surveying latency candidates for any PyVista+VTK app, always check whether the existing `skimage.measure.marching_cubes` call can be replaced with VTK's threaded Flying Edges via `pv.ImageData.contour(method='flying_edges')` before proposing higher-complexity solutions — it is already in the dep stack and is 1-2 OOM faster.
- **VTK-on-main-thread is a hard macOS constraint**: for any background-thread async mesh computation pattern in a PyVista/VTK app, only the mesh generation step (pure NumPy/scikit-image/PyVista filter operations) can move off-thread; `plotter.add_mesh()` and `plotter.render()` must remain on the main thread (macOS Cocoa + VTK GL context). napari/superqt's `@thread_worker` with signal-based main-thread handoff is the canonical implementation.
