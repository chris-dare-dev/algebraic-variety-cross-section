# Research Brief — realtime-variety-render-e5b (agent-a: MATH / PER-GENERATOR FIELD-EXPRESSION TRANSCRIPTION lens)

**Milestone:** Numba JIT field-evaluation kernel **v1**. Extend the `@njit(parallel=True, cache=True)` field-eval kernel pattern shipped in **e5 v0** (`_fermat_field_kernel` + `_enriques_fig1_field_kernel`) to the **9 remaining implicit generators** in `surfaces.py`: `kummer_surface`, `enriques_figure_2/3/4`, `calabi_yau_dwork`, `fano_klein_cubic / segre_cubic / two_quadrics / sextic_double_solid`.

---

## 1. TL;DR

For each of the 9 generators, write a per-voxel `@njit(parallel=True, cache=True)` kernel that is a **term-by-term operator-order-identical transcription** of the existing NumPy meshgrid + broadcast-polynomial + `np.clip` block — `prange` outer `i`, plain `range` `j/k`, scalar `g[i]/g[j]/g[k]`, every `**N` written as explicit multiplies (`x*x` not `x**2`, `x4*x` not `x**5`), and `np.clip(F, lo, hi)` folded as a per-voxel `if val < lo: val = lo; elif val > hi: val = hi`. **Main transcription risk:** Dwork's `**5` and Segre's `**3` are the only powers the e5 v0 templates never exercised — Numba would internally lower them, but explicit-multiply preserves IEEE-754 op-order parity with the NumPy reference at `rtol=atol=1e-9` and avoids ULP drift on arm64. **Fallback:** if any one of the 9 kernels surfaces an unexpected Numba issue, ship the other 8 and leave that one on its current NumPy path — the 9 kernels are independent (no cross-kernel state, separate `cache=True` `.nbi/.nbc` files per function).

## 5. Alternatives considered

- **Single generic `_polynomial_field_kernel(g, coeffs, out)` parameterized by a coefficient table.** Rejected: hides per-generator math behind a generic interface, breaks the term-by-term transcription discipline, complicates the reference functions in tests (would need parallel parameterization, doubling drift surface).
- **Lift `np.clip` outside the kernel (keep it as a separate NumPy pass).** Rejected per e5 lessons 152 + 159: re-introduces an `n³` temp allocation, defeats fusion, AND would require the test reference to also clip outside — fine, but the e5 v0 chose folded-in clip and v1 stays consistent.
- **`@guvectorize` or `@vectorize` instead of `@njit(parallel=True) + prange`.** Rejected: e5 v0 chose explicit-loop kernels; v1 stays consistent. Same machinery → §8.20 single-flight reasoning carries over unchanged.
- **Cache the JITs eagerly at import (startup warm-up).** Rejected per e5 spike §4/§8: unnecessary; the e4 worker absorbs first-touch JIT cost off the GUI thread; `cache=True` makes it once-per-machine. Eager warm-up is dead code.
- **Skip `fano_two_quadrics` (coarse_n=0).** Rejected: full-resolution release renders still benefit from kernel speedup; per-kernel JIT cost is identical to any other kernel; no engineering reason to opt out.
- **Hard-code Kummer's `sqrt(2.0)` as a Python float literal inside the kernel (`1.4142135623730951`) instead of passing as a scalar arg.** Functionally equivalent; e5 template style passes scalars in. Either is acceptable — implementer's choice.
- **Add `n = int(round(n))` to enriques_figure_2/3/4 in this milestone.** Defensive AI-8 hygiene — matches the `enriques_figure_1` pattern at `surfaces.py:614`. Currently figs 2/3/4 use plain `n: int` kwargs (no current callers pass float), so it's not strictly needed. Recommend ADD it (zero risk, one line, future-proofs against the same callers that the e5 brief flagged for fig 1).

## 6. Risks and unknowns

- **R1 — Dwork's `**5` is the only v1 kernel using a power not already in the e5 v0 templates.** Mitigation: explicit-multiply transcription (`x2 = x*x; x4 = x2*x2; x5 = x4*x`) — the explicit form pins the per-voxel op sequence and matches NumPy's `np.power` chained-multiply lowering. The `rtol=atol=1e-9` test catches any residual drift. **Same applies to Segre's `**3` (`x*x*x`) and sextic double solid's `**6` (already written as `X2*X2*X2` in the NumPy reference — copy that pattern verbatim into the kernel).**

- **R2 — Operator-order drift on multi-factor products** (Kummer's `lam*p*q*r*s`, Enriques fig 4's `4*t0*t1*t2`). Numba's per-voxel scalar evaluation order can differ from NumPy's broadcast-temporary chain (NumPy materializes `p_*q_` then `(p_*q_)*r_` then `((p_*q_)*r_)*s_` left-to-right; Numba may reorder a 4-way product). Mitigation: write the kernel with **EXPLICIT left-to-right grouping** matching the NumPy reference — `((p_*q_)*r_)*s_` or `(p_*q_)*(r_*s_)` — then write the NumPy reference function in the test using the SAME grouping. If the test fails at the chosen grouping, switch grouping in BOTH places (the contract is "kernel ≡ reference," not "kernel ≡ original broadcast"). Tolerance `rtol=atol=1e-9` absorbs ULP-level differences when grouping is consistent.

- **R3 — `np.clip(F, lo, hi)` JIT-ability.** Numba supports both `np.clip` (array form) and scalar `min(hi, max(lo, x))` / `if/elif` form. v1 uses the e5 `if val < lo: val = lo; elif val > hi: val = hi` form for every kernel — proven, fused, no temp allocation. NONE.

- **R4 — RuntimeWarning preservation across kernel migration** (Dwork ψ≈1, two_quadrics ε<0.08). Both `warnings.warn(...)` calls fire in the GENERATOR (`surfaces.py:986-990` Dwork; `:1138-1142` two_quadrics) BEFORE the field block. Verified by line-position read. Kernel is pure compute and never warns. The warnings continue to surface via `MeshWorker.run`'s `warnings.catch_warnings(record=True)` (`render_worker.py:196`) → `MeshResult.warning_text`. **AI-14 preserved.** No code-relocation risk; the warning lines stay where they are.

- **R5 — Kummer ValueErrors preservation.** `abs(mu_squared - 3.0) < 1e-6` (pole) at `surfaces.py:546-547` and `mu_squared <= 1.0/3.0` (no zero set) at `:548-551` fire BEFORE `lam` is computed (`:552`) and BEFORE the field block (`:558`). **Both stay in the generator** — kernel never sees an invalid `mu_squared` or `lam`. AI-14 preserved.

- **R6 — `_marching_cubes_to_polydata` AI-14 ValueError guards** are downstream of every kernel call (pre-contour at `surfaces.py:226-231`, post-contour at `:256-260`). Untouched by v1.

- **R7 — Numeric-equivalence tolerance on arm64.** Same risk as e5 R3 — Apple Silicon NEON may produce ULP-level differences vs x86 SSE/AVX. `rtol=atol=1e-9` is the proven tolerance (e5 v0 passed all probes at this tolerance on x86; the macOS arm64 gate stays open per e5 spike §7 as a pre-ship checklist, not an e5b blocker).

- **R8 — `workqueue` single-flight constraint (CONTEXT.md §8.20).** The constraint is "no two `@njit(parallel=True)` kernels can run concurrently process-wide" — already satisfied by `_computing` single-flight at `app.py:613` and `_pending_render`/`_pending_is_coarse` queue-latest at `app.py:228, 662-672`. v1 stacks 9 more kernels but does NOT widen the constraint. CONTEXT.md §8.20 unchanged. **The `_computing` guard scales for free.**

- **R9 — Cold-cache JIT stacking cost** (~2.7-3.6 s spread across 9 first-touches). Each ~300-400 ms compile is absorbed inside that surface's typical-generate window (Kummer/Enriques baseline ≈450 ms). The user only pays the compile on the FIRST click of each surface, and `cache=True` persists each kernel's compiled blob to `__pycache__/_*_field_kernel*.nbi/.nbc` (already covered by `.gitignore`'s `__pycache__/` entry). Per-machine, not per-process. Not a blocker.

- **R10 — Coarse-LOD interaction at coarse_n** (e4b composition). 8 of 9 new kernels run at coarse_n during drag (Kummer=100, Enriques 2/3/4=80, Dwork=100, Klein/Segre/sexticDS=80); `fano_two_quadrics` (coarse_n=0) runs only at full release. The kernel is invoked from inside `_marching_cubes_to_polydata` which is downstream of any specific `n` value (`params["n"] = surface.coarse_n` injection at `app.py:709`). Composes mechanically. **No app/worker/registry change** in e5b.

- **R11 — AI conflict scan.** AI-1: numba already sanctioned (compute dep per e5). AI-2: tests are pure NumPy + Numba (no Qt). AI-3: off-screen verification only. AI-6: implicit pipeline only; kernels feed `_marching_cubes_to_polydata`; Hanson parametric trio untouched. AI-8: `int(round(n))` coercions preserved in generators; kernels see only float `g` axis + float scalar params. AI-14: every ValueError pre-check + RuntimeWarning preserved BEFORE the kernel call. **No new AI conflicts.**
